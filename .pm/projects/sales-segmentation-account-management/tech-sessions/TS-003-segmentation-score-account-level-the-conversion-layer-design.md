decisions:
  - THREE LAYERS STAY DISTINCT. Segment (who they are) / Score (how good a prospect — potential, blind to spend) / Account-level (how we steward them). Account-level is the missing third layer and is NEVER silently derived from the other two. It also sits BESIDE — never replaces — the existing tierID (Diamond/Gold/Silver/Bronze value-grade) and accountType (Cash/Credit billing).
  - "THE ENGINE = a two-track Value×Potential grid, both axes in £/yr. POTENTIAL (X) = a static, anti-leakage curve from the score (never moved by spend). ACTUAL (Y) = realised hire-only revenue from ya_contracts. Levels: low actual+low potential → System-only (default floor); low actual+high potential → Incubation (the ~311 white-whale nursery); high actual → Account; high actual+high potential+repeat cadence → Strategic. Movement is VERTICAL (realised value climbs toward the customer's own fixed potential ceiling); share-of-wallet = actual÷potential; white-whale gap = vertical distance to the diagonal. This resolves Ben's 'move toward top-right' — it's 'up toward the ceiling', not lateral."
  - "NEW vs EXISTING is the two-track split. EXISTING (has REALISED hire history — a delivered contract, not just a future-dated booking) runs the full ladder on realised value. NEW (potential only) collapses onto the X-axis and is HARD-CAPPED at Incubation — a customer cannot be born Account/Strategic, they GRADUATE as real revenue accrues. Key fix: a forward-dated booking does NOT flip a new customer to 'existing' (that defeated the cap); it fires committed_fwd → big-fish alert + senior fast-track ownership NOW, but the durable level stays Incubation until revenue is realised. The single £60k one-off wedding routes to the 'Lifetime' follow-up process, NOT Strategic (Strategic now requires txn_count_12m >= 2 repeat cadence)."
  - "THE CONVERSION = a suggest→confirm state machine, not an algorithm that decides. System writes suggested_level only; a human sets confirmed_level (effective_level = COALESCE(confirmed, suggested)). States: suggested → proposed → owner_assigned → requirements_met → confirmed (+ rejected/demote). Each level has an ownership GATE before it is 'real': System-only → named pool steward (no individual salesID); Incubation → customerType-aware data capture (Corporate resolves companyType; Personal/no-domain uses the same set MINUS companyType so it can actually be satisfied); Account → semi-bespoke plan + manager approval; Strategic → bespoke plan + manager sign-off + a HARD per-senior-AM capacity cap (because ~311 candidates ≫ headcount). Transfer windows = quarterly review worklist (suggested≠confirmed, lapsed high-tier, split-ownership, disproven-whale). Big-fish = real-time manager alert on arrival."
  - "SEGMENT (95% missing) MODULATES, NEVER GATES a level — no rule depends on companyType, so the gap can't stall an assignment; segment is a fit-multiplier on potential (unknown ×1.0, never penalise missing data) and a Strategic tie-breaker. Enrichment is on-demand: resolving companyType becomes a requirement of the Incubation gate, so the engine itself becomes the segment-backfill engine. COMPETITORS are NOT excluded: a new customer_sales_scores.classification ENUM('mainstream','trade','disqualified') routes trade/sub-hire peers to a trade lane (System-only + trade pricing, or Account if high-volume) regardless of history; only genuinely not-events businesses are disqualified to System-only. GRAIN: the unit of stewardship is the economic account = email domain, rolled onto a deterministic 'home row' (merge primary, else highest-lifetime same-domain row) to avoid N-fold inflation across same-domain contacts; ownership conflicts (≥2 AMs on one domain) are FLAGGED, never silently overwritten."
  - "MODEL REFINED to a confidence-weighted blend (supersedes the hard new/existing two-track). Scoring+segmentation is UNIVERSAL (the 528 scored are mostly existing customers). Proposed level = blend: expected_value = alpha*realised_value + (1-alpha)*potential_value, where alpha in [0,1] is an evidence/confidence weight from delivered-order count + tenure + recency. New customer alpha~0 (proposal driven by score+segment + a weighted dab of initial quote value); established repeat alpha~0.8-0.9 (history dominates, score nudges). The 'born cannot be Strategic' rule becomes a consequence of alpha~0 + the realised-value gate, not a separate track. The potential-minus-realised GAP still drives Incubation/white-whale. ALL parameters (curve anchors, segment multipliers, alpha function, initial-quote weight beta, band cuts, white-whale SoW threshold, Strategic cap) live in a named/versioned PARAMETER SET with a what-if simulation screen that recomputes proposed levels over the scored base in-memory and shows the distribution shift + diff-vs-live before promoting a set to live. Design addendum in progress."
  - "ADD QUOTED VALUE as a third signal — revealed demand. A customer who quotes a lot but we are not winning still signals potential value (and a conversion problem worth sales attention). Quoted value sits between the cold web-lookup potential (they engaged) and realised won revenue (no money changed hands), so it feeds the POTENTIAL side of the blend: potential_value combines modelled (web score + segment) AND revealed (annualised deduped quoted demand); when revealed demand exists it is trusted over the web guess. This makes 'quotes lots / wins little' surface as a big potential-minus-realised GAP = white-whale / priority Incubation + a low win-rate (realised/quoted) conversion-problem flag (ties to the stopped-quotes / system-followup theme). CRITICAL: must DEDUPLICATE quotes before summing — same customer quoting multiple times around the same date for the SAME event (different item combos) is ONE opportunity, not N. Dedup key must include the HIRE/EVENT date (same created date + same hire date = duplicate variants -> collapse to one representative value e.g. converted-one-else-max; same created date + DIFFERENT hire dates = distinct opportunities -> keep). Naive SUM(quotesValue) massively overstates demand. Dedup applies equally to the last-300-new-quotes validation cohort."
  - "EMPIRICAL CALIBRATION from sandbox (2026-06-26), grounds the blend thresholds. (1) DEDUP: last-12m active+converted quotes = 27,393 rows -> 23,395 opportunities by (customer + event-date) = only ~14.6% inflation (moderate, not 2x); the copiedQuote flag carries £0 so it is UNUSED in practice -> dedup MUST be date-based (customer + hireStartDate +/-2 days, the existing UsefulFunctions convention), never the flag. (2) REALISED hire value distribution (trailing 12m, per customer): <£2k = 4,639 custs (85.6%), £2k-8k = 611, £8k-20k = 116, £20k-40k = 33, £40k+ = 22 (0.4%); ~5,421 active customers, ~£8.8M total. So band cuts of Account>=£8k (~171 custs) and Strategic>=£40k (~22 custs) are staffable as-is. Top 0.4% (£40k+) = ~18% of value; the <£2k mass = ~30% of value (so System-only still matters — justifies the steward/nurture investment). (3) The realised top is TINY (22 strategic, 171 account) -> growth must come from the white-whale/quoted-demand pool, which is exactly why the blend (quoted demand as potential) is needed, not a realised-only model."
  - "QUOTED-VALUE / WIN-RATE signal VALIDATED on real data. Among >=3-opportunity customers, win-rate ranges 0.03-1.00 at similar quoted values -> strongly discriminating. Canonical conversion-problem case: customer 8701 = 132 opportunities, 4 won (3% win rate), £69.6k quoted, ~£2k realised -> a realised-only model dumps them in System-only, but the blend (quoted demand -> potential) correctly elevates them to a high-priority Incubation / conversion-problem flag with a huge potential-minus-realised GAP. Other clear cases: 52903 (0.13, £53k), 5393 (0.25, £73k), 5194 (0.33, £105k), 59797 (0.00, £39k). Directionally ~£20M quoted vs ~£8.8M won hire in 12m => a large unconverted-demand prize the conversion lens targets. Use 8701 as the worked example in the addendum."
